# Ingestion Layer (Span — folder/photo/Gmail upload, presigned encrypted storage, dedup/idempotency, consent, queue handoff to pre-analytics)

## Overview
The ingestion layer is the front door for all PHI entering Span. It accepts lab reports through three paths — (a) bulk folder/multi-file upload, (b) iOS VisionKit photo capture of physical records, (c) Gmail OAuth fetch of lab-report attachments — and lands every artifact, byte-identical, in region-locked encrypted object storage, then enqueues exactly one parse job per *new* artifact for the pre-analytics pipeline. It is deliberately "dumb and durable": it does NOT parse, OCR, or interpret. Its only jobs are to (1) authenticate the user and capture/verify lawful-basis consent (GDPR Art.6/9 + DPDP verifiable consent) BEFORE any byte is stored, (2) pin both *inference geo* and *storage geo* to the user's region (India -> ap-south-1 Mumbai, EU -> eu-central-1 Frankfurt), (3) move bytes from device straight to object storage via short-lived presigned PUT URLs (bytes never transit the app server), (4) deduplicate via content hashing so re-uploads and Gmail re-fetches don't create duplicate work or duplicate measurements, (5) guarantee idempotency so client retries and at-least-once queue delivery never double-process, (6) drive a clear multi-stage progress UX (uploading -> queued -> parsing -> needs-review/done), and (7) hand off cleanly to pre-analytics with a typed job contract.\n\nGrounding in the existing data: the current single-patient corpus is 1,536 measurements / 159 parameters built from ~59 source files across 21 distinct lab labels (Healthians 710, CLUMAX 103, Thyrocare 94, Tata 1mg 43, plus 16 others, and 165 measurements with no lab attributed). That heterogeneity — many labs, varied PDF layouts, scanned photos, missing lab attribution — is exactly what the ingestion layer must normalize into a uniform stream of artifacts. Each artifact ultimately fans out into the existing flat Measurement rows (date, parameter, parameter_raw, category, value, value_text, unit, ref_low, ref_high, ref_text, flag, lab, sources[]), now keyed by user_id + report_id, with loinc_code added as the canonical cross-lab identifier. The ingestion layer owns the lifecycle from raw bytes to "a parse job is enqueued"; pre-analytics owns everything after.

## Architecture
DATA FLOW (all three paths converge on one ingestion-job state machine):\n\n  iOS SwiftUI app (region pinned at signup; PHI never in iCloud/CloudKit)\n   |\n   |-- (a) BULK FOLDER UPLOAD: Files.app / DocumentPicker multi-select (PDF/JPEG/PNG/HEIC) -> compute SHA-256 client-side per file -> POST /v1/ingestion/intents (batch) -> server returns N presigned PUT URLs (or 'duplicate' verdicts) -> client PUTs each file directly to S3 -> POST /v1/ingestion/{id}/complete\n   |\n   |-- (b) PHOTO CAPTURE: VNDocumentCameraViewController (on-device edge-detect + perspective de-skew + enhance, Notes-app quality) -> render pages to JPEG/PDF ON DEVICE (data minimization: only the cleaned doc leaves the phone) -> SHA-256 -> same intent->presign->PUT->complete path as (a)\n   |\n   |-- (c) GMAIL FETCH: user taps 'Import from Gmail' -> OAuth (gmail.readonly restricted scope, PKCE) -> backend Gmail worker runs SERVER-SIDE query filtered to known lab senders + has:attachment filename:pdf -> for each candidate, fetch attachment bytes -> SHA-256 -> de-dup check -> stream into S3 via the SAME internal intent path (no presign; server already holds bytes in-region)\n   v\n  API EDGE (FastAPI, region-routed; Aurora PostgreSQL system of record, RLS by user_id)\n   |  - AuthN: Sign in with Apple (primary). Gmail OAuth is a *data-access* grant, NOT primary login.\n   |  - Consent gate: every intent rejected (409 CONSENT_REQUIRED) unless an active consent row covers scope='ingestion' + the path used.\n   |  - Residency router: resolves user.region -> {storage_bucket, kms_key, inference_geo}. One region per data principal, never a global bucket.\n   |  - Dedup: content_sha256 unique per (user_id) -> existing artifact short-circuits (idempotent 'already have this').\n   v\n  REGION-LOCKED OBJECT STORAGE (S3 ap-south-1 / eu-central-1; MinIO on-prem fallback)\n   |  - SSE-KMS per-object (CMEK), TLS in transit, Object Lock (governance) + legal-hold capable, lifecycle rules (raw original retained per policy, transient renders expire).\n   |  - Key layout: s3://span-phi-{region}/u/{user_id}/raw/{artifact_id}.{ext}  (+ /pages/ for VisionKit page images)\n   v\n  INGESTION-JOB STATE MACHINE (Postgres-backed) + DURABLE QUEUE (SQS FIFO per region, MessageDeduplicationId=artifact_id)\n   |  states: intent_created -> uploading -> uploaded(verified) -> enqueued -> [pre-analytics owns] parsing -> needs_review|extracted -> committed | failed | duplicate | quarantined\n   |  On verified upload (S3 ETag/hash match + virus/MIME scan pass) the server transactionally writes job=enqueued AND emits ONE SQS message. Outbox pattern: message row written in same DB txn as state change, a relay publishes to SQS, so exactly-one-effective enqueue even under crash.\n   v\n  HANDOFF -> PRE-ANALYTICS (Document AI Form Parser/OCR in-region + Gemini structured extraction on Vertex asia-south1 / europe-west3; Claude on Bedrock EU only for EU/de-identified)\n      The queue message is the contract (see APIs). Pre-analytics consumes, sets parsing, runs OCR+LLM extraction -> standardized JSON -> writes measurements + updates artifact -> emits ingestion.parse.completed which the app polls/streams for progress.\n\nASCII — the convergent pipeline:\n\n  [Folder] [Photo/VisionKit] [Gmail OAuth worker]\n      \\          |                 /\n       \\         |                /\n        v        v               v\n     +------------------------------------+\n     |  POST /ingestion/intents (consent  |\n     |  gate + residency router + dedup)  |\n     +------------------------------------+\n        | presigned PUT (a,b)   | internal put (c)\n        v                       v\n     +------------------------------------+\n     |  S3 (region-locked, SSE-KMS)       |\n     +------------------------------------+\n        | /complete -> verify hash+MIME+AV\n        v\n     +------------------------------------+\n     |  ingestion_job FSM (Postgres)      |\n     |  outbox -> SQS FIFO (dedup=artifact)|\n     +------------------------------------+\n        |  ingestion.parse.requested (msg)\n        v\n     [ PRE-ANALYTICS: OCR -> Gemini -> LOINC/units -> confidence tiers -> measurements ]\n        |  ingestion.parse.completed (event)\n        v\n     [ App progress UX + organ-system analysis ]\n\nIDEMPOTENCY & DEDUP (three independent layers):\n 1. Client retry safety: every mutating call carries Idempotency-Key (UUID per logical action); server stores key->result for 24h, replays identical response.\n 2. Content dedup: content_sha256 over raw bytes; UNIQUE(user_id, content_sha256). A re-uploaded identical file resolves to the existing artifact_id with status 'duplicate' — no new job, no new measurements. (Mirrors how the current corpus collapses identical (date,parameter,value,unit) across duplicate report files and preserves filenames in sources[].)\n 3. Queue dedup: SQS FIFO MessageDeduplicationId = artifact_id; consumers are idempotent on artifact_id so at-least-once delivery never double-writes measurements (upsert keyed on (user_id, report_id, parameter, date)).\n\nNEAR-DUPLICATE handling (Gmail re-sends, photo of a PDF already uploaded): exact-hash misses these, so a soft dedup signal is computed downstream (lab + collection_date + parameter set) and surfaced as a merge suggestion in review, not auto-merged.

## APIs / interfaces
REST (FastAPI, versioned /v1, region-routed by user; all calls require SiwA bearer token; mutating calls require Idempotency-Key header):\n\nPOST /v1/ingestion/intents        -- batch begin (folder & photo). Body: { items: IngestionIntentRequest[], consent_ack: bool }\n  -> 200 { results: IngestionIntentResponse[] }    (per-item 'new' w/ presigned PUT, or 'duplicate' w/ existing_status)\n  -> 409 CONSENT_REQUIRED if no active 'ingestion' consent\n  -> 413 if byte_size over cap; 415 unsupported MIME\n\nPUT  <presigned-s3-url>            -- client uploads bytes directly to region S3 (server never sees bytes for a,b)\n\nPOST /v1/ingestion/{job_id}/complete -- client signals upload done. Server HEADs object, verifies ETag/size, recomputes/zerifies hash on a sampled/async basis, runs AV+MIME sniff.\n  -> 200 { job_id, status:'enqueued' }   (transactionally writes status + outbox row -> SQS)\n  -> 422 UPLOAD_MISMATCH (hash/size/MIME mismatch -> job=failed, object deleted)\n\nGET  /v1/ingestion/jobs?status=&since=   -- progress polling list (drives UX). Returns [{job_id, artifact_id, source, original_filename, status, progress_pct, error_code, updated_at}]\nGET  /v1/ingestion/jobs/{job_id}          -- single job detail\nGET  /v1/ingestion/stream                 -- optional SSE/WebSocket push of job status transitions (fallback to polling)\n\nGMAIL PATH:\nPOST /v1/integrations/gmail/connect       -- starts OAuth (PKCE); scope=https://www.googleapis.com/auth/gmail.readonly. Writes consent(scope='gmail_readonly', method='oauth_grant').\nPOST /v1/integrations/gmail/sync          -- triggers server-side scan. Server query: from:(known lab senders) has:attachment filename:pdf newer_than:Xy, paginated via history_id. For each PDF attachment: dedup by hash -> if new, stream bytes -> S3 (internal intent, source='gmail') -> job enqueued. Returns { found, new, duplicate, skipped }.\nDELETE /v1/integrations/gmail             -- revoke token + Google revocation endpoint; consent.withdrawn_at set.\n\nCONSENT / RIGHTS (one pipeline satisfies App Store 5.1.1, GDPR erasure/portability, DPDP):\nPOST /v1/consent                          -- record/withdraw a consent (verifiable). Body: { scope, granted, policy_version }\nGET  /v1/me/export                        -- portability: signed bundle of artifacts + measurements\nDELETE /v1/me                             -- account+data deletion: tombstone, async purge S3 + DB rows + revoke Gmail; audit-logged.\n\nQUEUE CONTRACT (ingestion -> pre-analytics), SQS FIFO, MessageGroupId=user_id, MessageDeduplicationId=artifact_id:\nTopic ingestion.parse.requested {\n  schema_version:1, job_id, artifact_id, user_id, region,\n  inference_geo:'in'|'eu',                 -- pin: India -> Vertex asia-south1; EU -> Vertex europe-west3 or Bedrock eu-central-1\n  storage:{ bucket, key, kms_key_id }, source, mime_type, page_count?,\n  hint:{ sender_domain?, lab_guess?, original_filename }, content_sha256\n}\nEVENT BACK (pre-analytics -> ingestion/app), topic ingestion.parse.completed {\n  schema_version:1, job_id, artifact_id, user_id,\n  result:'extracted'|'needs_review'|'failed', report_id?,\n  measurements_written:int, low_confidence_count:int, error_code?\n}\nConsumer rules: idempotent on artifact_id; measurements upserted on (user_id,report_id,parameter,date); on failure -> DLQ after N attempts -> job=failed with error_code; poison artifacts -> quarantined (kept in S3, no measurements).\n\nHANDOFF SEMANTICS: ingestion's responsibility ENDS at 'enqueued' + a durable parse.requested message that points at the exact in-region encrypted object and carries the residency pins. Pre-analytics never re-derives region or re-fetches from the device; it reads only the S3 key + inference_geo in the message, keeping both residency controls (storage geo, inference geo) locked end-to-end.

## Tech choices
- iOS capture: VisionKit VNDocumentCameraViewController wrapped via UIViewControllerRepresentable — native on-device edge-detect/de-skew/enhance (Notes quality), zero server cost, and crucially keeps raw camera frames on device (data minimization); only the cleaned JPEG/PDF leaves the phone.
- Bulk + photo upload transport: presigned S3 PUT with per-object SSE-KMS — bytes go device->bucket directly, never through the FastAPI app server, shrinking the PHI blast radius and the residency surface to exactly one bucket per region.
- Object storage: AWS S3 ap-south-1 (Mumbai) for India principals, eu-central-1 (Frankfurt) for EU; MinIO as the S3-API-compatible on-prem fallback if DPDP localization tightens. Object Lock + lifecycle for retention; one region per data principal, no global bucket.
- System of record: AWS Aurora PostgreSQL under signed BAA, region-pinned (ap-south-1 / eu-central-1), Row-Level Security by user_id — chosen over Supabase because Supabase has no India region and was blocked in India (Feb 2026), disqualifying it for Indian PHI.
- Durable queue: SQS FIFO per region with MessageDeduplicationId=artifact_id + a transactional-outbox relay — gives exactly-once-effective enqueue and natural idempotency, in-region (queue data stays in the user's AWS region).
- Backend: Python/FastAPI — best fit alongside the downstream multimodal-LLM parse and data-science longevity models; async by default for the upload-orchestration and Gmail-fetch I/O.
- Auth: Sign in with Apple as primary login (limits to name+email, private-relay email) — also pre-empts Guideline 4.8 if Google is ever used for login. Gmail OAuth is treated strictly as a data-access grant, decoupled from identity.
- Gmail scope: restricted gmail.readonly (metadata scope cannot read attachments) — accept the Google CASA Tier 2/3 assessment (~$15k-$75k, re-passed every 12 months, weeks lead time); minimize footprint with server-side sender+PDF filtering before any download. Residency note: token + fetched bytes are processed in-region and bytes land in the region bucket; phase Gmail AFTER folder/photo to defer CASA cost.
- Inference residency pins carried in the job message, not hardcoded in pre-analytics: India -> Vertex asia-south1 (Gemini + Document AI, both regional, BAA-eligible, the only single-vendor true-India stack); EU -> Vertex europe-west3 or Claude on Bedrock eu-central-1 with In-Region routing + ZDR. Claude-in-India-on-Bedrock is global cross-region inference, so it is excluded from the India path.
- Residency modeled as two independent controls (storage geo + inference geo) locked per user-region at signup, mirroring Anthropic's own model — the ingestion layer is where both are first stamped onto every artifact and every queue message.

## Risks
- Gmail CASA assessment is a recurring annual cost ($15k-$75k) with weeks of lead time and re-verification every 12 months; if it lapses, the gmail.readonly grant must be disabled or the app risks Google verification revocation. Mitigation: ship folder+photo first, gate Gmail behind a feature flag, calendar the re-assessment, and minimize scope footprint (server-side sender+PDF filter) to ease the audit.
- DPDP future data-localization notification: the Central Government may mandate localization for sensitive categories and health data is sensitive; a non-India storage or inference hop could become non-compliant overnight. Mitigation: India users already fully pinned to ap-south-1 + Vertex asia-south1; keep the MinIO/self-hosted-OCR fallback designed-in so we can go fully in-country/air-gapped.
- Claude-in-India-on-Bedrock looks regional (Mumbai) but actually runs global cross-region inference, so accidentally routing Indian PHI to Claude/Bedrock would break India residency. Mitigation: inference_geo pin in the queue message + a hard policy guard that rejects any India-region artifact routed to a non-asia-south1 endpoint.
- Presigned-URL leakage or over-long TTL could expose a PHI object. Mitigation: short TTL (e.g. 5 min), single-use semantics, URL bound to exact key+content-length+SSE-KMS headers, TLS only, and the object is unreadable without the regional KMS key.
- Near-duplicate (not byte-identical) re-ingestion — a Gmail re-send re-encoded, or a phone photo of an already-uploaded PDF — slips past SHA-256 dedup and could create duplicate measurements. Mitigation: soft-dedup signal (lab+collection_date+parameter-set) surfaced as a merge suggestion in human review; never auto-merge.
- Malicious/poison uploads (malware PDF, decompression bomb, non-lab image) consuming pipeline cost or pivoting. Mitigation: MIME sniff + AV scan + size/page caps at /complete BEFORE enqueue; failures quarantined, never auto-fed to the LLM.
- Background/large bulk uploads (decades of reports, like the existing 59-file corpus) interrupted by app backgrounding or flaky network. Mitigation: per-file independent jobs (no all-or-nothing batch), resumable via idempotency keys, URLSession background uploads, and a foreground catch-up reconciliation of job status on app launch.
- Consent-before-storage ordering bug could land PHI without lawful basis. Mitigation: consent gate enforced server-side at the intent endpoint (409 CONSENT_REQUIRED), not client-side; verifiable consent row written and audit-logged before any presigned URL is issued.
- OCR/LLM region BAA constraint tightening (a processor pulled from a region) could strand the parse step. Mitigation: self-hosted OCR+VLM fallback (PaddleOCR PP-StructureV3 / Surya + local VLM) designed in from day one so extraction can run inside our own EU/India infra.

## Phased build
- Phase 0 — Foundations & residency spine: region-pinned Aurora (ap-south-1 + eu-central-1) with RLS, region S3 buckets with SSE-KMS + Object Lock + lifecycle, Sign in with Apple, consents + audit_log tables, and the residency router (user.region -> bucket/kms/inference_geo). No ingestion yet — just the rails both storage and inference pins ride on.
- Phase 1 — Bulk folder upload end-to-end: DocumentPicker multi-select, client SHA-256, POST /intents with consent gate + dedup, presigned PUT, /complete with hash+MIME+AV verify, ingestion_jobs FSM, outbox->SQS, and a stub pre-analytics consumer that just acks. Ships a real, durable, idempotent upload path against the simplest source. Validates the convergent state machine with the messiest real data (the existing 59-file multi-lab corpus as test fixtures).
- Phase 2 — VisionKit photo capture: VNDocumentCameraViewController SwiftUI wrapper, on-device de-skew/enhance, render-to-PDF, reuse the exact intent->presign->complete path (source='photo', page_count, captured_at). Plus the full progress UX (uploading/queued/parsing/needs-review/done) with polling + optional SSE stream and foreground catch-up reconciliation.
- Phase 3 — Real pre-analytics handoff: wire the live parse pipeline (Document AI + Gemini in-region per inference_geo) consuming ingestion.parse.requested, writing measurements (LOINC + units + confidence tiers), emitting parse.completed; idempotent upsert on (user,report,parameter,date); DLQ/quarantine; near-duplicate merge suggestions in review. End-to-end: a photographed Tata 1mg report becomes plotted measurements.
- Phase 4 — Gmail OAuth fetch (gated): gmail.readonly OAuth (PKCE), server-side sender+PDF filtered sync with history_id incremental cursor, dedup-aware fetch into the same pipeline, revoke flow. Run/clear the Google CASA assessment in parallel; keep behind a feature flag until the Letter of Assessment is in hand.
- Phase 5 — Rights, hardening, on-prem fallback: in-app account deletion + data export wired to one erasure/portability pipeline (App Store 5.1.1 + GDPR + DPDP), append-only audit completeness, and the MinIO + self-hosted OCR/VLM fallback proven for a fully in-country/air-gapped deployment in case DPDP localization tightens.

## Open questions
- Retention policy specifics: how long do we keep the raw original artifact in S3 after successful extraction (DPDP storage-limitation + GDPR minimization vs. user value of re-parsing on model upgrades)? Need a concrete per-region retention window and whether users can opt to delete raw-but-keep-measurements.
- Region assignment authority: is region fixed at signup from a declared residency, or inferred (locale/SIM/Apple region)? And is cross-region migration (EU user moves data to India or vice versa) ever supported, or hard-locked? This determines whether region is immutable on users.
- Gmail phasing/budget sign-off: confirm CASA budget and timeline, and whether Gmail launches India-first (Tata 1mg/Healthians/Thyrocare senders) or is deferred entirely past MVP given cost/lead time.
- Lab-sender allowlist source of truth: who curates the known-lab sender-domain list (the corpus already shows 21 labs incl. Healthians, CLUMAX, Thyrocare, Tata 1mg, RV Metropolis, Redcliffe, SRL, Manipal/KMC)? Static config, admin-managed table, or learned from accepted extractions?
- Confidence-tier thresholds and who reviews: what self-consistency agreement cutoffs map to auto/review/flagged, and is the human-in-the-loop reviewer the user themselves, an internal ops team, or both — and does that reviewer sit in-region for residency?
- Max bulk batch size / file caps: upper bounds on files-per-batch, bytes-per-file, and pages-per-document for AV/decompression-bomb safety and cost control.
- Does HealthKit-derived data (steps/sleep) flow through this same ingestion-job FSM as a synthetic 'device' artifact, or via a separate lighter sync path? Research suggests treating wearables as a 'device' lab source uniformly, which would argue for the former.
- On-prem trigger: is the MinIO + self-hosted OCR/VLM fallback a contingency only, or a committed deliverable for a specific enterprise/government Indian customer that would change priority and timeline?
